# prefix 選択基準

## 使える prefix

| Prefix | 意味 | 使用場面 |
| --- | --- | --- |
| `feat_` | 新機能の追加 | 新しい操作導線や対応範囲を増やす |
| `fix__` | バグ修正 | 既存の期待動作の不具合を直す |
| `docs_` | ドキュメントのみの変更 | README、設計書、コメント整備 |
| `style` | 意味に影響しない変更 | フォーマット、空白、並び順 |
| `refa_` | リファクタリング | 内部構造の整理、責務分割、命名改善 |
| `perf_` | パフォーマンス改善 | 速度、メモリ、I/O の改善 |
| `test_` | テストの追加・修正 | 単体、結合、E2E テストの変更 |
| `chore` | ツールやビルドの変更 | 依存関係、CI、ビルド設定、スクリプト |

## docs repo の扱い

- docs repo では `docs_` を原則使わない
- docs repo ではドキュメント更新自体が自明なため、変更の性質に応じて別 prefix を選ぶ
- 手順や内容の誤り修正は `fix__`、情報設計や章構成の整理は `refa_`、運用補助や AI agent 向け整備は `chore`、整形だけなら `style` を候補にする

## `feat_`

- ユーザー価値のある新機能、画面、API、動作追加に使う
- これまで無かった操作導線、対応プラットフォーム、入力手段、設定項目の追加に使う
- PC にしか無かった操作をモバイルへ広げるような変更も含める
- プロジェクトの動作に直接的な影響を与えない変更には使わない
- skills、prompts、agents、README、設計資料などの更新には使わない

## `fix__`

- 既に提供しているはずの挙動が壊れているときに使う
- 仕様とずれている、回帰した、エラーになる場合に使う
- 新しい導線の追加ではなく、壊れた挙動の復旧を優先する

## `docs_`

- README、設計書、コメント整備などに使う
- 振る舞いを変えない文書更新に使う
- docs repo の commit には使わない

## `style`

- フォーマット、空白、lint 修正、並び順調整に使う
- 意味や挙動に影響しない変更に限る

## `refa_`

- 内部構造の整理、責務分割、命名改善、重複除去に使う
- 機能追加やバグ修正を含まない変更に使う

## `perf_`

- 速度、メモリ効率、描画性能、I/O 効率の改善に使う
- 待ち時間や再取得回数の削減が主目的のときに使う

## `test_`

- 単体テスト、結合テスト、E2E テストの追加・修正に使う
- 検証コードの整備が主目的のときに使う

## `chore`

- 依存関係、CI、ビルド設定、開発ツール、スクリプトの変更に使う
- ソースコード本体の振る舞いを直接変えない変更に使う
- skills、prompts、agents など AI agent 向けファイルの更新に使う
