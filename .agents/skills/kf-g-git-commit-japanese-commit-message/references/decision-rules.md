# 判断ルール

## 複数変更がある場合

- 主目的を優先する
- 新機能の追加に伴ってテストも増えるなら `feat_` を優先する
- バグ修正のために内部整理したなら `fix__` を優先する

## repo の性質を先に見る

- docs repo では `docs_` を第一候補にしない
- docs repo ではその repo の主目的がドキュメント格納であることを前提に、変更の性質で prefix を決める
- docs repo での誤記や手順不備の修正は `fix__`、情報設計や章構成の整理は `refa_`、運用補助や AI agent 向け整備は `chore`、整形だけなら `style` を優先する

## 外から見える効果を優先する

- 新しくできるようになることが増えるなら `feat_` を選ぶ
- 本来できるはずだったことを元に戻すなら `fix__` を選ぶ
- 未対応だった操作や環境への対応は、原則 `feat_` を選ぶ

## `feat_` は直接的な動作変更に限定する

- 実行時挙動、ユーザー操作、API 応答、表示結果に直接影響する変更だけに使う
- ドキュメント、skills、prompts、agents、設計メモなどの更新には使わない
- 直接的な動作変更が無いなら `refa_`、`chore`、`docs_`、`style` などを再検討する

## subject はカテゴリ先頭で書く

- subject の先頭に必ず `<カテゴリ>_` を置く
- カテゴリは変更対象を短く表す語にする
- `skills_`、`prompts_`、`認証_`、`設定画面_` のように、対象を先に置いてから内容を書く

## 既存コミットとの整合

- 直近 10 commit の各 1 行目を確認する
- 既存の commit message で使われている単語や文章形式が今回の差分に適しているなら、type_ や subject をできるだけ寄せる
- 関連する commit だと後から分かりやすくするため、無理のない範囲で表現を流用する

## 情報が不足している場合

- 分かる範囲で最も無理のない案を出す
- 勝手に細部を作らない
- 説明が必要なときだけ、前提を短く添える
- 曖昧なときは、外から見える効果を基準にする
