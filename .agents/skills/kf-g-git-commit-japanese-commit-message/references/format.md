# 出力形式と基本ルール

## 出力形式

- 1行目は `<type_>: <カテゴリ>_<subject>` にする
- 2行目は必ず空行にする
- 3行目以降に本文を書く
- 本文は箇条書きで書く
- 少なくとも `- 概要` と `- Why:` を含める

## 本文の書き方

- `概要` には何を変えたかを書く
- `Why` にはなぜこの対応を行なったかを書く
- 1行で収まるなら短くまとめる
- 複数点ある場合や長くなる場合は、配下の箇条書きへ分ける
- 理由と背景は混ぜすぎない

## subject の書き方

- 日本語で書く
- 先頭に必ず `<カテゴリ>_` を付ける
- カテゴリは変更対象を短く表す語にする
- カテゴリ例は `skills_`、`prompts_`、`認証_`、`設定画面_` など
- カテゴリの後ろに変更の主目的を書く
- 変更の主目的を1つに絞る
- 冗長な前置きは入れない
- 文末の句点は付けない
- 体言止めを優先する
- 敬語は使わない
- 「修正対応」「対応を実施」のような曖昧表現を避ける
- 何を変えたかが分かる語にする

### 良い例

- `fix__: 検索_検索結果0件時のローディング終了不備を修正`
- `refa_: 認証_サービス層分離`
- `docs_: README_開発環境構築の前提条件を追記`

### 避ける例

- `fix__: いろいろ修正`
- `fix__: 検索結果0件時のローディング終了不備を修正`
- `chore: 対応`
- `feat_: skills_プロンプト改善`

## 文体

- 一文を短く保つ
- 接続詞や前置きを最小限にする
- 体言止めを多用する
- 全体を端的に保つ
