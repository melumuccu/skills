# 作成手順

## 手順

1. 直近 10 commit の各 1 行目を確認し、既存の type_ や subject に寄せられないか検討する
1. 対象の repo が docs repo かどうかを確認する
1. 変更内容から主目的を 1 つに要約する
1. 8 種類の prefix から最も適切なものを選ぶ
1. 主目的が新機能か、不具合修正かを切り分ける
1. プロジェクトの動作に直接的な影響が無いなら `feat_` を除外する
1. docs repo なら `docs_` を原則除外し、変更の性質に応じて prefix を選び直す
1. 変更対象を短く表すカテゴリを決める
1. `<カテゴリ>_` から始まる日本語 subject にする
1. 本文に `概要` と `Why` を書く
1. 長くなる箇所は次の箇条書きへ分ける
1. 体言止め中心で整える
1. 1行目を `<type_>: <カテゴリ>_<subject>` の形にする
1. 2行目を空行にする
1. prefix が 5 文字固定か確認する

## 本文テンプレート

```text
<type_>: <カテゴリ>_<subject (日本語)>

- 概要:
  - 何を変えたか
  - 必要なら次の要点
- Why:
  - なぜこの対応を行なったか
  - 必要なら背景や制約
```

## 書き方の要点

- カテゴリは `skills_`、`prompts_`、`認証_`、`設定画面_` のように簡潔にする
- prefix とカテゴリは別に考える
- `feat_` を選ぶ前に、実行時挙動が直接変わるかを確認する
- 一文が長くなりそうなら、次の箇条書きへ分ける
- 1 つの箇条書きには 1 論点だけを書く
- 本文も日本語で簡潔に書く
- 長文より短い箇条書きを優先する
